## 🎭 場景一：自癒與自動擴縮 (Self-Healing & HPA)

**📂 涉及檔案**：`k8s/02-rollout.yaml`, `k8s/03-service.yaml`, `k8s/04-hpa.yaml` **展示內容**：展示 EKS 如何像「活的生物」一樣自我管理。

------

### ⚠️ 環境大掃除（重複演示前必做）

```powershell
kubectl delete rollout ecommerce-backend --ignore-not-found
kubectl delete svc ecommerce-svc --ignore-not-found
kubectl delete hpa ecommerce-hpa --ignore-not-found
kubectl delete pod load-generator --ignore-not-found

# 確認清除完成
kubectl get pods
kubectl get svc
kubectl get hpa
# 預期結果：均回傳 No resources found in default namespace.
```

------

### Step 1｜部署應用與服務

```powershell
kubectl apply -f k8s/02-rollout.yaml
kubectl apply -f k8s/03-service.yaml
```

------

### Step 2｜等待外部網址就緒 (Address)

```powershell
# 等待 EXTERNAL-IP 從 <pending> 變為實際網址
kubectl get svc ecommerce-svc -w
```

> **意義**：這是 AWS 自動為您建立的 **Network Load Balancer (NLB)**。NLB 運作於 OSI 第四層（傳輸層），相比舊世代的 CLB，提供極高的效能與固定 IP 處理能力，是現代 EKS 生產環境的首選。
>
> 取得網址後（格式為 `k8s-...ap-northeast-1.amazonaws.com`），將其貼到瀏覽器，確認可以看到 Podinfo 介面，證明流量已從 Internet 成功進入 EKS。
>
> **⚠️ 技術補充**：本專案使用 `aws-load-balancer-nlb-target-type: instance` 模式，無需額外安裝 AWS Load Balancer Controller，即可建立真正的 NLB，降低架構複雜度。

------

### Step 3｜觀察 Rollout 狀態

```powershell
kubectl get rollout ecommerce-backend -w
```

> **意義**：以 **SRE 監控視角** 即時觀察 Pod 的健康狀態（包含期望數、目前數與可用數），確認所有副本均已 Ready 後再進行下一步。

------

### Step 4｜視覺化驗證自癒能力 (Self-Healing) 🩺

```powershell
# 刪除所有 Pod，模擬服務崩潰
kubectl delete pod -l app=ecommerce-backend

# 即時觀察 Pod 自動重生
kubectl get pods -l app=ecommerce-backend -w
```

> **現象**：終端機畫面會呈現完整的三階段自癒流程：
>
> ```
> Pending → ContainerCreating → Running
> ```
>
> 整個重生過程約 **3～5 秒**內完成。
>
> **⚠️ 演示說明口訣**：「這裡看到舊 Pod 顯示的是 `Completed` 而非 `Terminating`，這是因為我們使用了 **Argo Rollouts** 來管理 Pod 生命週期。Argo Rollouts 實現了**優雅終止（Graceful Termination）**，Pod 不是被強制殺死，而是等它處理完手上所有進行中的請求後才退場，確保**零請求丟失**，這是生產環境的標準做法，也是比原生 Deployment 更嚴謹的地方。」
>
> **意義**：這展現了 K8s 的「期望狀態 (Desired State)」管理。身為維運者，我們不需要半夜起床重啟服務，K8s 的 ReplicaSet 會自動監控並確保系統始終擁有正確數量的副本。
>
> **瀏覽器驗證**：切換到瀏覽器畫面，Podinfo 頁面底部的 `Served by` 欄位已更新為新 Pod 的名稱，且頁面全程**無中斷**，證明自癒過程對使用者完全透明。

### Step 5｜啟動 HPA

```powershell
kubectl apply -f k8s/04-hpa.yaml
```

### Step 6｜驅動 HPA：壓力測試 ⚡

```powershell
# 啟動強化版壓測：同時發出多個併發請求，確保 CPU 壓力足以觸發長滿 10 個 Pod
kubectl run -i --tty load-generator --rm --image=busybox:1.28 --restart=Never -- /bin/sh -c "while true; do wget -q -O- http://ecommerce-svc & wget -q -O- http://ecommerce-svc & wget -q -O- http://ecommerce-svc & wget -q -O- http://ecommerce-svc & wait; done"
```

------

### Step 7｜觀察 HPA 即時擴容（開另一個視窗執行）📊

```powershell
# 觀察 TARGETS 百分比上升 (超過 30% 即觸發) 與 REPLICAS 增加
kubectl get hpa ecommerce-hpa -w

# 同時觀察 Pod 如何從 2 個自動擴展為 10 個
kubectl get pods -l app=ecommerce-backend -w
```

> **預期現象（對照實際結果）**：
>
> | 階段       | CPU  | REPLICAS | 說明                   |
> | ---------- | ---- | -------- | ---------------------- |
> | 壓測前     | 2%   | 2        | 正常待機，低於門檻     |
> | 壓測開始   | 152% | 2        | CPU 暴衝，HPA 開始計算 |
> | 第一波擴容 | 239% | 6        | 每次最多 +4 個 Pod     |
> | 第二波擴容 | 198% | 10       | 達到 maxReplicas 上限  |
>
> **⚠️ 演示說明口訣**：「大家可以看到這張表，CPU 在壓測後衝到 152%，遠超我們設定的 30% 門檻，HPA 立刻觸發第一波擴容到 6 個 Pod，239% 還不夠撐，再擴到上限 10 個。整個過程全自動，沒有任何人工介入，從壓力感知到資源補充只花了不到 2 分鐘。」
>
> **AWS 觀察點 1 - EC2 Console**：進入 AWS Console → **EC2** → **執行中實例** → 選中 EKS Node → 點擊「監控 (Monitoring)」標籤，可看到 **CPU 使用率**折線圖同步飆升。
>
> **AWS 觀察點 2 - Auto Scaling Group**：進入 **EC2 → Auto Scaling Groups**，當 Pod 多到節點放不下時，`Cluster Autoscaler` 會觸發 ASG，「期望容量 (Desired Capacity)」自動增加並啟動新 EC2 實例。

### Step 8｜收尾清理 🛑

```powershell
# 1. 在壓測視窗按下 Ctrl + C 停止壓測

# 2. 確保壓測 Pod 已釋放
kubectl delete pod load-generator --ignore-not-found

# 3. 觀察 Pod 數量自動縮回 2 個 (約需等待 30 秒)
kubectl get pods -l app=ecommerce-backend -w

# 4. 刪除 HPA（場景二前置會明確刪除，提前清除避免干擾副本數）
kubectl delete hpa ecommerce-hpa --ignore-not-found

# 5. 刪除 Rollout 與 Service，釋放 AWS NLB
kubectl delete rollout ecommerce-backend --ignore-not-found
kubectl delete svc ecommerce-svc --ignore-not-found

# 6. 確認清除乾淨
kubectl get pods
kubectl get svc
kubectl get hpa
# 預期結果：三個指令均回傳 No resources found in default namespace.
```

### 🔍 技術原理深度解析

**① HPA 計算公式**

`Metrics Server` 採集 CPU 數據，HPA 根據以下公式決定副本數：

```
目標副本數 = ceil[目前副本數 × (目前CPU% ÷ 目標CPU%)]
```

**② HPA 震盪控制（Thrashing Prevention）**

> 「您可能會注意到我的 HPA 設定檔使用了 v2 版本的 `behavior` 區塊。在真實生產環境中，流量經常是突發且不穩定的。若沒有設定 `stabilizationWindowSeconds`（冷卻時間），Pod 會隨流量波動頻繁建立與刪除，這稱為『震盪 (Thrashing)』。透過精細設定此參數，可確保擴容迅速、縮容平穩，保護底層節點的穩定性。」

**③ 雙層擴展架構：公寓與住客**

> 「我們可以把 EKS 想像成『公寓與住客』的關係：
>
> - **Pod 擴展 (HPA)**：公寓裡突然湧入大量住客，需要開更多的『房間 (Pods)』。
> - **Node 擴展 (Cluster Autoscaler)**：住客多到整棟公寓都住滿時，`Cluster Autoscaler` 偵測到有住客在排隊（Pending Pods），便自動向 AWS 申請一台新的 **EC2 實例 (Node)**，等同蓋了一層新樓。
> - 這種由內而外（Pod → Node）的連動，確保系統在流量高峰時，從應用層到硬體層實現完全自動化的彈性增長。」

**④ 架構價值總結**

> 「這套架構實現了從應用程式壓力到雲端硬體資源的完全自動化連動，大幅降低維運成本並提升業務穩定性。」
